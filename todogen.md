# Phase 6 — TODO Generation

Read ALL prior phase outputs before generating the TODO.
Pull every FAIL and PARTIAL check from Phases 3, 4, and 5.
Pull every CRITICAL, HIGH, and MEDIUM finding from Phase 3.

---

## Rules

- Do not hardcode any tool name, file path, or value — derive everything from prior phase outputs
- Every item describes exactly one thing to create, change, or add
- Every item names the actual file path, script name, tool, or config key involved
- No item says "improve", "consider", or "add better" — it says exactly what to do
- Effort is estimated based on what was found, not a generic guess
- Do not include items for checks that passed
- Do not include placeholder examples — every item is real

---

## Item format

```
- [ ] [EFFORT] What to create or change, naming the actual file/tool/value
```

Effort scale:
- XS = under 1 hour
- S = 1–3 hours
- M = half day
- L = full day
- XL = 2–3 days

---

## How to generate each section

### 🔴 Immediate
Source: Phase 3 CRITICAL findings
For each CRITICAL finding: write one or more items describing the exact
remediation steps, referencing the file paths and values found in the audit.

### 🟠 Workflow Foundation
Source: Phase 4 FAIL and PARTIAL checks
For each failed check: write one item describing what to create or add,
naming the exact file, script, recipe name, or config based on the
stack detected in Phase 1.

### 🟡 Pipeline & Delivery
Source: Phase 5 FAIL and PARTIAL checks
For each failed check: write one item describing what to add or change
in the CI/CD pipeline, version tracking mechanism, migration tooling,
rollback recipe, or infrastructure codification — using the actual
tools and file paths detected in Phase 1 and Phase 2.

### 🔵 Security Hardening
Source: Phase 3 HIGH and MEDIUM security findings
For each finding: write one item describing the exact change,
referencing the actual file, function, config key, or middleware
where the fix belongs.

### 🟢 Observability & Reporting
Source: Phase 3 Reporting gaps (HIGH and MEDIUM)
For each finding: write one item describing the exact addition,
referencing the actual service, file, or config where it belongs.

### ⚪ Deferred
Source: Phase 3 LOW findings and Phase 4/5 PARTIAL checks
that are non-urgent given the current project stage.

## Task granularity rules

- XS tasks are written as-is
- S tasks are written as-is
- M tasks must be broken into subtasks:
  ```
  - [ ] [M] Parent task description
    - [ ] [XS] Subtask 1
    - [ ] [XS] Subtask 2
    - [ ] [S]  Subtask 3
  ```
- L and XL tasks must always be broken into subtasks — never left as a single item
- A subtask must be completable in one sitting without depending on another subtask
  in the same parent (except where order is strictly required — note it explicitly)

## Additional sources for TODO items

Beyond FAIL/PARTIAL checks from Phases 4–5 and findings from Phase 3, also generate
items for:

**From Phase 1 section 1.16 (Questionable architecture decisions):**
- For each external Docker volume blocking startup: add task to replace with
  in-container volume or eliminate the volume dependency

  add task to add that alternative to the local Docker Compose stack
- For each frontend analytics tool with a self-hostable alternative:
  add task under ⚪ Deferred (unless cost is a MEDIUM+ finding from Phase 3)

- For each absent AI config file that is relevant to the detected tooling:
  add task to create it with minimum viable content
  (project overview, how to run, architecture summary, conventions)

**From Phase 4 section 4.6 (E2E complexity):**
- If e2e setup is undocumented: add task as M broken into XS subtasks
- If no smoke/full split exists: add task to create the split
- Do not add e2e tasks as single L items — always break them down
